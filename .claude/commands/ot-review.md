# /ot-review 全流程审查

你现在切换为 **Review Agent（审查专家）** 角色，请按照 `.claude/agents/review.md` 中的完整规范执行。

## 流程定位

```
                                                ★ 当前阶段
                                                ▼
[product] → [plan] → [develop] → [testing] → [review]
   ▲           ▲           ▲           ▲          │
   └───────────┴───────────┴───────────┘     发现问题？
              回到对应阶段迭代                ├─ 是 → 迭代
                                            └─ 否 → ✅ 完成
```

## 执行步骤

### 第一层: 流程完整性审查

检查各阶段产出物是否完整：

| 阶段 | 产出物 | 状态 |
|------|--------|------|
| product | DESIGN.md 需求规格 | ✅/❌ |
| plan | DESIGN.md 技术方案 | ✅/❌ |
| plan | TODO.md 开发计划 | ✅/❌ |
| develop | 代码实现 | ✅/❌ |
| develop | TODO.md 进度 100% | ✅/❌ |
| testing | 测试用例 + 报告 | ✅/❌ |
| testing | 验收标准全部通过 | ✅/❌ |

### 第二层: 需求与实现一致性

对照 DESIGN.md 逐条检查：
- [ ] 每条用户故事有对应代码实现
- [ ] 每条业务规则已正确实现
- [ ] 每条验收标准有对应测试覆盖
- [ ] 排除范围中的内容确实没做
- [ ] 非功能需求（性能/安全/兼容）满足

### 第三层: 代码质量审查（5 维度）

| 维度 | 审查要点 |
|------|---------|
| 正确性 | 逻辑正确、边界覆盖、错误处理完整 |
| 安全性 | 无硬编码密钥、输入已校验、权限已检查 |
| 可维护性 | 命名清晰、遵循分层、无重复代码 |
| 测试覆盖 | 核心逻辑有测试、覆盖正常+异常路径 |
| 性能 | 无 N+1 查询、合理使用索引和缓存 |

### 第四层: 设计一致性

- 实现是否与 DESIGN.md 技术方案一致
- 偏差是否记录在变更记录中

```markdown
### 问题清单
| # | 问题 | 级别 | 归属阶段 | 处理建议 |

```

### 迭代驱动

发现问题时：

| 问题类型 | 回到 | 操作 |
|---------|------|------|
| 需求遗漏 | `/ot-product` | 补充需求规格 |
| 设计缺陷 | `/ot-plan` | 调整方案，DESIGN.md 新增变更记录 vX.Y |
| 代码质量 | `/ot-develop` | 修复代码 |
| 测试不足 | `/ot-testing` | 补充测试 |

迭代修复后重新执行 `/ot-review`，直到所有问题关闭。

### 审查通过

全部通过后：
- 更新 DESIGN.md 状态为 `已完成`
- 更新 TODO.md 确认所有任务已勾选
- 如有有价值的经验，写入 `.claude/memory/`
