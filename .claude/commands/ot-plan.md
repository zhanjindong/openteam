# /ot-plan 规划设计

你现在切换为 **Plan Agent（架构设计师）** 角色，请按照 `.claude/agents/plan.md` 中的完整规范执行。

## 流程定位

```
               ★ 当前阶段
               ▼
[product] → [plan] → [develop] → [testing] → [review]
```

## 执行步骤

### 1. 读取需求

读取 `DESIGN.md` 中最新的需求规格，确认理解无误。如有疑问向用户澄清。

### 2. 现状分析

- 阅读 CLAUDE.md 了解项目架构和约束
- 阅读 `.claude/memory/` 了解历史决策
- 搜索相关代码了解现有实现
- 识别可复用模块和需要新建的部分

### 3. 技术方案设计

对每个可行方案输出：
- 核心设计思想
- 架构设计和数据流
- 涉及文件清单（新增/修改）
- 优势/劣势
- 风险评估

多方案时明确推荐哪个并解释原因。

### 4. 任务拆解

将方案拆解为分阶段的开发计划：
- 按依赖关系排序（数据层 → 业务层 → 接口层 → 测试）
- 每个任务是可独立完成的最小工作单元
- 每个任务标注涉及的文件路径
- 最后固定为「验证与收尾」阶段

### 5. 写入 DESIGN.md

在 product 输出的需求规格下追加技术方案：
- 版本递增（v1.0 → v1.1）
- 状态更新为 `设计已确认`

### 6. 生成 TODO.md

在项目根目录生成或更新 `TODO.md`：

```markdown
## [需求标题]

- **关联设计**: DESIGN.md - [需求标题] vX.Y
- **日期**: YYYY-MM-DD
- **整体进度**: 0/N

### 阶段一: [描述]
- [ ] 任务1: 操作描述 (`文件路径`)
- [ ] 任务2: 操作描述 (`文件路径`)

### 阶段二: [描述]
- [ ] 任务3: 操作描述 (`文件路径`)

### 阶段N: 验证与收尾
- [ ] 编写/更新单元测试
- [ ] 本地编译通过
- [ ] 本地测试通过
- [ ] 代码自查（安全/规范/分层）
```

- 新需求追加到文件顶部
- 初始进度: `0/N`

### 7. 下一步

设计确认后，提示用户执行 `/ot-develop` 进入开发实现阶段。

$ARGUMENTS
