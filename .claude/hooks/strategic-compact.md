# Strategic Compact — 上下文压缩协议指南

## 概述

Claude Code 的对话存在上下文窗口限制，当对话过长时系统会自动压缩历史消息。Strategic Compact 协议定义了一套结构化的压缩策略，确保压缩后最关键的信息被保留，避免"压缩失忆"导致的重复工作和错误决策。

## 核心问题

**无策略压缩的典型症状:**
- 压缩后忘记之前的设计决策，重复讨论同一问题
- 丢失正在进行的任务上下文，需要重新解释需求
- 忘记已经尝试过的失败方案，重复踩坑
- 丢失文件修改记录，不知道哪些文件已经改过

## 压缩优先级模型

上下文信息按 4 个层级分类，压缩时从 L4 开始丢弃，逐级向上：

```
┌─────────────────────────────────┐
│  L1 — 绝不丢弃 (核心锚点)        │  ← 压缩后必须完整保留
├─────────────────────────────────┤
│  L2 — 尽量保留 (工作状态)         │  ← 压缩为摘要
├─────────────────────────────────┤
│  L3 — 可压缩   (过程细节)         │  ← 压缩为一句话或丢弃
├─────────────────────────────────┤
│  L4 — 可丢弃   (临时信息)         │  ← 直接丢弃
└─────────────────────────────────┘
```

### L1 — 绝不丢弃（核心锚点）

这些信息丢失会导致后续工作方向性错误：

| 内容 | 说明 |
|------|------|
| 当前任务目标 | 用户最终想要实现什么 |
| 已确认的设计方案 | DESIGN.md 中已确认的方案要点 |
| 关键约束和限制 | 用户明确提出的"不要做什么" |
| 已修改的文件清单 | 本次会话中改过哪些文件，改了什么 |
| 当前任务进度 | TODO.md 中哪些完成了，下一步是什么 |
| 未解决的阻塞问题 | 遇到的还没解决的问题 |

### L2 — 尽量保留（工作状态）

这些信息有助于高效继续工作，压缩时保留摘要：

| 内容 | 压缩方式 |
|------|---------|
| 代码修改的具体 diff | 压缩为 → "文件X: 新增了方法A，修改了方法B" |
| 讨论过的备选方案 | 压缩为 → "考虑过方案A/B/C，选了B，因为..." |
| 关键代码片段 | 压缩为 → "关键逻辑在 文件:行号，使用了XX模式" |
| 调试过程的关键发现 | 压缩为 → "根因是XX，通过YY方式发现" |

### L3 — 可压缩（过程细节）

这些信息是过程性的，可以大幅压缩：

| 内容 | 处理方式 |
|------|---------|
| 完整的文件读取内容 | 丢弃原文，只保留 "已读取文件X，包含Y个方法" |
| 命令执行的完整输出 | 丢弃原文，只保留 "执行XX命令，结果: 成功/失败+关键信息" |
| 搜索和探索的过程 | 丢弃过程，只保留 "在XX位置找到了YY" |
| 多轮讨论的中间消息 | 丢弃中间讨论，只保留最终结论 |

### L4 — 可丢弃（临时信息）

这些信息在压缩时直接丢弃：

- 问候语和闲聊
- 工具调用的原始 JSON
- 重复的错误信息（只保留最终的解决结论）
- 已放弃的方案的详细讨论过程
- 格式调整、拼写修正等琐碎改动

## 压缩触发机制

### 主动压缩（推荐）

当感知到以下信号时，Claude 应主动执行压缩：

1. **对话超过 30 轮**: 主动整理一次上下文摘要
2. **切换任务**: 开始新任务前，将旧任务压缩为摘要
3. **关键里程碑**: 完成 TODO.md 中的一个阶段后，压缩该阶段的过程

### 主动压缩的执行步骤

```
Step 1: 生成压缩摘要
         ↓
Step 2: 将 L1 信息完整列出
         ↓
Step 3: 将 L2 信息压缩为摘要
         ↓
Step 4: 将有价值的发现写入 .claude/memory/（跨会话保存）
         ↓
Step 5: 输出压缩摘要供后续上下文使用
```

### 压缩摘要模板

当执行压缩时，生成以下结构化摘要：

```markdown
---
## 📋 上下文压缩摘要

### 当前任务
- **目标**: [一句话描述]
- **进度**: [M/N] — 参见 TODO.md
- **当前阶段**: [正在做什么]

### 已完成的工作
- [文件A]: [做了什么改动]
- [文件B]: [做了什么改动]
- ...

### 关键决策
- [决策1]: [选了什么，为什么]
- [决策2]: [选了什么，为什么]

### 待解决的问题
- [问题1]: [状态和下一步]

### 关键约束（用户明确要求）
- [约束1]
- [约束2]
---
```

## 与其他机制的配合

Strategic Compact 不是孤立运作的，它与整个配置体系协同：

```
                    ┌──────────────┐
                    │  DESIGN.md   │  ← 设计决策的完整版（不受压缩影响）
                    └──────┬───────┘
                           │
┌──────────────┐    ┌──────┴───────┐    ┌──────────────┐
│   TODO.md    │◄───│   会话上下文   │───►│ .claude/     │
│  任务进度     │    │  (会被压缩)   │    │ memory/      │
└──────────────┘    └──────┬───────┘    │ 跨会话持久化  │
                           │            └──────────────┘
                    ┌──────┴───────┐
                    │  Strategic   │
                    │  Compact     │
                    │  压缩协议     │
                    └──────────────┘
```

| 外部文件 | 压缩时的作用 |
|---------|-------------|
| `DESIGN.md` | 设计决策的"真相源"，压缩时只需引用，不需在上下文中重复 |
| `TODO.md` | 任务进度的"真相源"，压缩时只需记录当前步骤，完整进度查文件 |
| `.claude/memory/` | 跨会话知识的持久化层，压缩时将有价值的发现写入 memory |
| `rules/` | 硬性规则不受压缩影响，始终通过文件加载 |

## 最佳实践

1. **DESIGN.md 和 TODO.md 就是最好的压缩保险**: 关键信息已持久化到文件，即使上下文被压缩也能随时读取恢复
2. **长任务主动分段**: 复杂任务每完成一个阶段，主动执行一次压缩，不要等系统强制压缩
3. **关键信息外置**: 发现重要信息时及时写入 memory/，不要只依赖上下文记忆
4. **压缩后自检**: 压缩后检查 L1 信息是否完整，如有丢失立即从 DESIGN.md / TODO.md 恢复
