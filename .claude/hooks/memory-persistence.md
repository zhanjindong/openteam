# Memory Persistence — 会话记忆持久化指南

## 概述

Claude Code 的每次会话是独立的，上下文在会话结束后丢失。Memory Persistence 机制通过会话生命周期钩子，将关键上下文持久化到 `.claude/memory/` 目录，实现跨会话的知识沉淀。

## 核心原理

```
会话启动                    会话进行中                     会话结束
   │                           │                            │
   ▼                           ▼                            ▼
加载历史记忆              实时积累关键上下文            触发 Stop 钩子
(memory-load.sh)         (Claude 自动跟踪)           (memory-save.sh)
   │                           │                            │
   ▼                           ▼                            ▼
读取 .claude/memory/      跟踪: 关键决策/发现/         写入 .claude/memory/
下的所有 .md 文件          模式/问题/解决方案            更新对应 .md 文件
```

## 目录结构

```
.claude/memory/
├── project-context.md       # 项目上下文（架构决策、技术选型、核心约束）
├── active-tasks.md          # 当前进行中的任务和状态
├── decisions.md             # 关键技术决策记录（为什么选 A 不选 B）
├── patterns.md              # 发现的代码模式和惯例（项目特有的写法）
├── issues.md                # 已知问题和排坑经验
└── contacts.md              # 关键模块负责人和协作信息
```

## 记忆分类与优先级

| 优先级 | 分类 | 说明 | 示例 |
|--------|------|------|------|
| P0-必记 | 关键决策 | 影响架构的技术决策和原因 | "选择 WebSocket 而非轮询，因为需要实时推送" |
| P0-必记 | 踩坑经验 | 排查耗时超过 10 分钟的问题及解决方案 | "MyBatis-Plus Join 查询需要手动指定 resultType" |
| P1-应记 | 代码模式 | 项目特有的、非通用的编码模式 | "所有 Client 类使用 WebFlux 而非 RestTemplate" |
| P1-应记 | 任务状态 | 未完成任务的当前进度和下一步 | "用户模块重构完成 3/7，下一步处理权限校验" |
| P2-可记 | 偏好发现 | 团队成员的编码偏好和习惯 | "该项目偏好 Builder 模式构造复杂对象" |
| 不记 | 临时信息 | 调试日志、临时变量、探索性尝试 | 不保存 |

## 记忆文件格式规范

每个记忆文件使用统一格式：

```markdown
# [分类名称]

> 最后更新: YYYY-MM-DD HH:mm

## [主题]

- **日期**: YYYY-MM-DD
- **上下文**: [什么场景下产生的]
- **内容**: [具体信息]
- **关联文件**: [相关文件路径]（如有）
```

## 会话启动 — 加载记忆

会话启动时，通过 CLAUDE.md 中的指令引导 Claude 加载记忆：

> 在 CLAUDE.md 中添加以下内容:

```markdown
## 会话初始化

每次会话启动时，如果 `.claude/memory/` 目录存在，请先读取其中所有 .md 文件，
快速了解项目上下文、进行中的任务、已知问题和关键决策，避免重复询问已有信息。
```

## 会话结束 — 保存记忆

通过 Stop 钩子触发 `memory-save.sh`，提醒 Claude 将本次会话的关键信息持久化。

### 保存触发条件

不是每次会话都需要保存，仅当以下情况发生时才需写入：

1. 做出了新的技术决策
2. 发现并解决了非显而易见的问题
3. 发现了项目特有的代码模式
4. 任务进度发生变化（开始、推进、完成）
5. 用户明确要求记住某些信息

### 保存原则

- **增量更新**: 不重写整个文件，只追加或更新变化的部分
- **去重**: 检查是否已有相同记忆，避免重复记录
- **精简**: 每条记忆控制在 3 行以内，只记核心信息
- **可操作**: 记忆内容应该对未来会话有实际参考价值
- **有上下文**: 记录产生这条记忆的背景，方便未来理解

## 记忆清理策略

记忆不是只增不减的，需要定期清理：

| 触发条件 | 清理动作 |
|---------|---------|
| 任务标记为已完成 | 从 `active-tasks.md` 移除，归档到 `decisions.md`（如有决策） |
| 记忆文件超过 100 行 | 合并相似条目，删除过时信息 |
| 问题已被代码修复 | 从 `issues.md` 中标记为已解决，保留经验部分 |
| 项目架构发生重大变更 | 更新 `project-context.md`，清理不再适用的模式 |

## 与其他机制的配合

| 机制 | 配合方式 |
|------|---------|
| DESIGN.md | 设计决策的详细版在 DESIGN.md，memory 中只保留一句话摘要和引用 |
| TODO.md | 任务进度在 TODO.md，memory/active-tasks.md 只记录任务概况和阻塞点 |
| rules/ | rules 是硬性规则，memory 记录规则背后的 "为什么" |
| strategic-compact | 压缩时优先保留 memory 中 P0 级别的信息 |
